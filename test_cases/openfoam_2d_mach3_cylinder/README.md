# Mach 3 Cylinder — OpenFOAM Test Case

This is the reference test case used to validate the gradient-based AMR
workflow in `final.py` and `all_run.sh`. It runs the 2D flow over a cold
cylinder at Mach 3 and 50 km altitude, using `rhoCentralFoam`, with three
AMR loops driven by the pressure-gradient sizing field.

## Physical setup

| Quantity | Value | Notes |
|---|---|---|
| Geometry | 2D cylinder | Radius 0.5 m, infinite cylinder (frontAndBack: empty) |
| Altitude | 50 km | Free-stream based on 1976 US Standard Atmosphere |
| Free-stream velocity `U_inf` | 990 m/s | Mach 3 at `T_inf` |
| Free-stream pressure `p_inf` | 79.78 Pa | |
| Free-stream temperature `T_inf` | 270.65 K | |
| Wall temperature `T_wall` | 800 K | Isothermal, no-slip |
| Solver | `rhoCentralFoam` | Compressible, density-based |
| Flux scheme | Kurganov with Minmod limiter | `system/fvSchemes` |
| AMR driver | `|grad(p)|` | `extraction_mode = gradient`, `gradient_field = p` |
| Sizing formula | `h = h_min + (h_max − h_min) / (1 + α·|∇p|/max|∇p|)` | `h_min=0.002`, `h_max=0.015`, `α=100` |
| Number of AMR loops | 3 | see `comet_result/output_amr{1,2,3}_result/` |

## Folder layout

The case folder contains only the case-specific inputs and the archived
results. The pipeline scripts (`all_run.sh`, `run_openfoam.sh`,
`run_openfoam_amr.sh`, `final.py`, `build_pvd_from_steps.sh`) are kept
once at the top of the repository and shared by every test case. The
case's `amr_pipeline.input` references those scripts with relative paths
(`../../run_openfoam.sh`, etc.).

| Path | What it is |
|---|---|
| `amr_pipeline.input` | Configuration for this case — paths to binaries, mesh and geometry filenames, extraction settings, and OpenFOAM solver settings. The single source of truth that both the orchestrator and the wrapper scripts read from. |
| `2d_cylinder.msh` | Initial uniform Gmsh mesh used by the first `rhoCentralFoam` run. |
| `2d_cylinder_amr.msh` | Final AMR mesh after the third loop. |
| `gmsh/2d_cylinder.geo` | Base geometry + mesh-sizing template for the uniform mesh. |
| `gmsh/2d_cylinder_amr.geo` | Geometry + `Merge "mfp.pos"; PostView` template for the AMR meshes. |
| `openfoam_case/0.orig/p`, `U`, `T` | Initial and boundary conditions. The wrapper script copies this to `0/` before each solver run. |
| `openfoam_case/system/controlDict` | Run control: solver endTime, write interval, `functionObjects`. |
| `openfoam_case/system/fvSchemes` | Numerical schemes (Kurganov flux, Minmod limiter). |
| `openfoam_case/system/fvSolution` | Linear-solver tolerances and relaxation. |
| `openfoam_case/constant/thermophysicalProperties` | Ideal-gas + Sutherland viscosity model. |
| `openfoam_case/constant/turbulenceProperties` | Laminar (no turbulence model). |
| `comet_result/output_normal1_result/` | Archived result from the first baseline run. |
| `comet_result/output_amr{1,2,3}_result/` | Archived results from each AMR iteration. |

After a run, the orchestrator additionally produces several files under
this directory: per-iteration mesh backups (`gmsh/2d_cylinder1.msh`,
`gmsh/2d_cylinder_amr1.msh`, `2d_cylinder_amr2.msh`, etc.), the
intermediate sizing field (`gmsh/mfp.pos`, `gmsh/all_data.csv`,
`gmsh/filtered_mfp.csv`), pipeline logs (`logs/`), and the OpenFOAM
working tree under `openfoam_case/` (`0/`, time-step directories,
`polyMesh/`, `VTK/`, `log.*`). These are regenerated by every run.

## How to reproduce

1. Install Gmsh (≥ 4.11), ParaView (with `pvpython`), and OpenFOAM v2406.
2. Open `amr_pipeline.input` and update the three absolute paths for your
   machine: `gmsh_bin`, `pvpython`, `foam_source`.
3. From this directory, run the orchestrator from the repository root:

   ```bash
   ../../all_run.sh
   ```

   The orchestrator auto-detects the current directory as the case base,
   so all output is written here. The OpenFOAM wrapper scripts at the
   repository root (`../../run_openfoam.sh` and
   `../../run_openfoam_amr.sh`) are invoked automatically through the
   `case_script` and `case_script_amr` keys in `amr_pipeline.input`.
4. Each loop writes a fresh archive under `comet_result/output_amr*_result/`.

To inspect the bundled results without running the solver:

```bash
paraview comet_result/output_amr3_result/field/auto_field.pvd
```

## Known limitations

- The mesh is generated by Gmsh as unstructured triangles. Raising the
  free-stream Mach number to 3.5 or above makes `rhoCentralFoam` crash at
  the bow shock on this mesh; Mach 3 is the practical ceiling with this
  setup.
- `amr_pipeline.input` contains absolute paths deliberately — they are
  meant to be edited once per machine, not generalised away, because
  that is how the upstream pipeline was designed to work.
